## 📖 תיאור הפרויקט

Remotix היא מערכת לשליטה מרחוק (Remote Desktop) מתקדמת. אפליקציית הלקוח מפותחת ב-C# ו-WPF, ומגובה בשרת ניהול מרכזי המפותח ב-C++. בשונה מפתרונות קיימים כמו TeamViewer, AnyDesk או Chrome Remote Desktop, המטרה הסופית של Remotix היא לאפשר שליטה סימולטנית של מספר משתמשים על מחשב אחד (Multi-User Control) - יכולת שלא קיימת כיום בפתרונות המסחריים המובילים.

### 🎯 מטרת הפרויקט
בחרנו לפתח את Remotix כי רצינו שילוב של אפליקציה ורשתות, וזאת אופציה שמשלבת את שניהם. מרבית הפתרונות הקיימים מגבילים את השליטה למשתמש אחד בכל רגע נתון, ואינם תומכים בשיתוף פעולה מרחוק בזמן אמת בין כמה גורמים.
המוטיבציה שלנו היא לפתח כלי קל לשימוש שיאפשר שליטה קבוצתית בצורה יצירתית - למשל לצורך:

- 👨‍🏫 **למידה פעילה** - מורים ותלמידים עובדים יחד
- 🛠️ **תמיכה טכנית משותפת** - פתרון בעיות בצוות
- 🎮 **הפעלת תוכנה באופן שיתופי** - עבודה משותפת על משחקים או תוכנות
  
### 👥 קהל יעד

הפרויקט מיועד לכל מי שמעוניין לשתף שליטה על מחשב מרוחק – באופן סימולטני - עם משתמשים נוספים. לדוגמה:
- אנשי תמיכה טכנית העובדים בצוותים
- מורים ותלמידים בסביבות לימוד מרוחקות
- משתמשים פרטיים המעוניינים להפעיל יחד תוכנה או משחק

**לא נדרש ידע מוקדם לשימוש בתוכנה** - הממשק נועד להיות אינטואיטיבי וקל לשימוש, החל ממסך ההתחברות ועד לשליטה עצמה.

## ✨ פיצ'רים מרכזיים

### ✅ קיים ופעיל

#### 1. 🖥️ שיתוף מסך מהיר ויעיל
* **לכידה:** Windows.Graphics.Capture API - לכידת מסך ברזולוציה מלאה
* **דחיסה:** H.264 (OpenH264) עם איכות מותאמת דינמית
* **שידור:** UDP Sockets עם Packet Fragmentation
* **ביצועים:** עד 30 FPS עם latency של 50-150ms

#### 2. 🎚️ Adaptive Quality System
מנגנון אינטליגנטי שמתאים את איכות השידור בזמן אמת על פי תנאי הרשת ורמת התנועה במסך:

**Motion Detection חכם:**
- 🟢 **Static** - תוכן סטטי (מסמכי טקסט, קוד)
- 🟡 **Low Motion** - תנועה מועטה (גלישה באינטרנט)
- 🟠 **Medium Motion** - תנועה בינונית (עבודה רגילה)
- 🔴 **High Motion** - תנועה אינטנסיבית (וידאו, משחקים)

**התאמות דינמיות:**
- **Bitrate:** 0.8-10 Mbps (מותאם אוטומטית)
- **FPS:** 10-30 FPS (משתנה לפי עומס)
- **Smart Response:** מגיב רק לשינויים משמעותיים

#### 3. 📦 Jitter Buffer מתקדם
מנגנון אגירה בסגנון TWS (True Wireless Stereo) המטפל בבעיות רשת:

**יכולות:**
- ⏱️ **Packet Buffering:** אגירת חבילות עד 150ms למזעור השפעת jitter
- 🔄 **Loss Recovery:** זיהוי אוטומטי של חבילות אבודות
- 📨 **I-Frame Request:** בקשה אוטומטית ל-I-Frame במקרה של packet loss קריטי
- 🔀 **Out-of-Order Handling:** טיפול בחבילות שמגיעות לא בסדר
- 📊 **Real-time Statistics:** מעקב אחר packet loss, recovery rate, duplicates

#### 4. 🖱️⌨️ שליטה מרחוק מלאה
שליטה בזמן אמת על מחשב מרוחק עם תחושת שליטה מקומית:

**Mouse Control:**
- תנועה חלקה (60 updates/sec)
- תמיכה בכל סוגי הלחיצות (Left/Right/Middle), גלילה (Mouse Wheel) ו-Drag & Drop.

**Keyboard Control:**
- תמיכה מלאה בכל המקשים וקיצורי מקלדת מורכבים (Modifier keys).

**מימוש טכני:**
- Win32 API - `SendInput()` עם Global Hooks. Latency: ~50-100ms.

#### 5. 🔒 מערכת חדרים, משתמשים והגדרות דינמיות
- **שרת מרכזי (C++):** מנהל הרשמות, התחברויות, וחדרים וירטואליים לשיתוף מרובה לקוחות.
- **לוח בקרה (Dashboard):** מסך יעודי ליצירת חדר, צפייה בחדרים זמינים והצטרפות מהירה.
- **הגדרות GUI:** שינוי כתובת השרת מתבצע מתוך האפליקציה ונשמר בקובץ `appsettings.json` (ללא צורך בקידוד קשיח).

#### 6. 🔄 התאוששות מתקלות אוטומטית
- **I-Frame Recovery:** בקשת I-Frame אוטומטית בעת packet loss משמעותי.
- **Connection Resilience:** המערכת ממשיכה לעבוד גם בתנאי רשת לא אידיאליים.
- **Graceful Degradation:** הורדת איכות אוטומטית לשמירה על חוויית משתמש.

### 🛠️ בתכנון

- **NAT Traversal:** חיבור מחוץ לרשת המקומית ללא Port Forwarding
- **שליטה סימולטנית:** מנוע "ממוצע תנועות" לשליטה של מספר משתמשים במקביל (התשתית בשרת קיימת, הפיתוח בלקוח מתמשך)
- **ניהול הרשאות:** תפקידים בחדר - Admin, Viewer, Controller
- **תת-חדרים והעברת קבצים**

## 🔧 ארכיטכטורה וטכנולוגיות

### מבנה הפרויקט

```text
Remotix/
├── Server/                          # שרת ניהול מרכזי (C++)
│   ├── Communicator.cpp/h           # ניהול שקעי TCP, האזנה ללקוחות
│   ├── RoomManager.cpp/h            # ניהול חדרים, רשימות משתתפים והצטרפות
│   ├── RequestHandlerFactory.cpp/h  # ניתוב בקשות הלקוח (Login, Signup, CreateRoom)
│   ├── SqliteDatabase.cpp/h         # גישה למסד הנתונים
│   └── WSAInitializer.cpp/h         # אתחול וניהול WinSock
│
├── Remotix_united/                  # אפליקציית הלקוח המאוחדת (C# WPF)
│   ├── AppSettings.cs               # ניהול הגדרות דינמי (appsettings.json)
│   ├── NetworkManager.cs            # תקשורת TCP מול השרת
│   ├── LoginWindow.xaml             # ממשק התחברות
│   ├── DashboardWindow.xaml         # ממשק ניהול חדרים (Dashboard)
│   ├── RoomWindow.xaml              # חלון השליטה/צפייה בתוך החדר
│   ├── SettingsWindow.xaml          # ממשק הגדרות רשת
│   └── (Media Handlers)             # מחלקות לכידת מסך, JitterBuffer, Win32 Hooks
│
└── Docs/                            # תיעוד ואפיון

```

### סטאק טכנולוגי

**שפות ופלטפורמות:**

* 🔷 **C# 10.0** - שפת הפיתוח הראשית ללקוח
* 🔶 **C++ 11/14** - שפת הפיתוח לשרת המרכזי
* 🎨 **WPF (Windows Presentation Foundation)** - ממשק משתמש
* 🗄️ **SQLite** - מסד הנתונים בשרת

**וידאו וגרפיקה:**

* 📹 **Windows.Graphics.Capture** - לכידת מסך
* 🎬 **SharpDX** - גישה ישירה לגרפיקה (Direct3D)
* 🗜️ **H264Sharp** - Wrapper ל-OpenH264

**תקשורת:**

* 🌐 **TCP Sockets** - לניהול משתמשים, חדרים ואימות מול השרת (פורט 12345)
* 🌐 **UDP Sockets** - לתקשורת P2P מהירה (שידור וידאו בפורט 12345, שליטה בפורט 12346)
* 📦 **JSON** - פורמט העברת נתונים

**קלט (Input Handling):**

* ⌨️ **Win32 API** - `SetWindowsHookEx()` ללכידת Global Hooks
* 🖱️ **SendInput()** - סימולציה של עכבר ומקלדת

## ⚠️ מגבלות ידועות

1. **🌐 רשת מקומית לוידאו בלבד**
* השרת מנהל חדרים מכל מקום, אך דורש חיבור ישיר בוידאו (אין עדיין NAT Traversal לוידאו)
* לא תומך בחיבור דרך האינטרנט ללא Port Forwarding.


2. **🪟 Windows בלבד**
* תלוי ב-Windows.Graphics.Capture API ו-Win32 API
* לא תומך ב-Linux/macOS


3. **📡 תקשורת UDP**
* חבילות אבודות עלולות לגרום לקפיצות בווידאו
* ה-Jitter Buffer ממזער את הבעיה אך לא פותר לחלוטין


4. **🎬 H.264 בלבד**
* תלוי ב-OpenH264 DLL ואין תמיכה בקודקים אחרים


5. **👤 משתמש יחיד בשליטה**
* כרגע תומך רק ב-1:1 (צופה אחד). שליטה סימולטנית נמצאת בתהליך עבודה.


6. **🔒 אבטחה**
* ישנה מערכת אימות (TCP) מול השרת, אך אין עדיין הצפנה לתעבורת הוידאו והשליטה ב-UDP.

## ⚙️ הגדרות והרצה

**המערכת עברה שדרוג ארכיטקטוני מלא! הקוד הקשיח (Hardcoding) של כתובות ה-IP הוסר כליל.**

### הפעלת שרת הניהול (Server)

1. קמפל והרץ את פרויקט ה-**Server** (ב-C++).
2. השרת יתחיל להאזין לבקשות לקוח (TCP) בפורט 12345.

### הפעלת הלקוח וחיבור (Remotix_united)

1. הפעל את אפליקציית הלקוח (WPF).
2. **הגדרת IP:** בחלון ההתחברות לחץ על סמל ההגדרות (⚙️). הקלד את כתובת ה-IP של המחשב עליו רץ השרת (לדוגמה `127.0.0.1` במקרה של הרצה מקומית). הכתובת תישמר ב-`appsettings.json`.
3. **התחברות:** הירשם או התחבר עם המשתמש שלך.
4. **Dashboard:** תועבר ללוח הבקרה. כעת תוכל ליצור חדר חדש או להצטרף לחדר פעיל מתוך הרשימה ולהתחיל בשיתוף.

### טיפים חשובים:

⚠️ **Firewall:** ודא ש-Windows Firewall מאפשר את פורטים 12345 (TCP/UDP) ו-12346 (UDP).
⚠️ **OpenH264.dll:** הקובץ חייב להיות בתיקיית ה-Build של אפליקציית הלקוח (למשל `bin/Debug/net6.0-windows/`).

## 🧪 בדיקות שבוצעו

### בדיקות ידניות:

✅ **ניהול משתמשים וחדרים:**

* התחברות והרשמה תקינות מול ה-DB בשרת.
* יצירת חדר, צפייה בחדרים זמינים ב-Dashboard והצטרפות עובדים כראוי.

✅ **לכידת מסך:**

* רזולוציה: 1920×1080, 1366×768, 1280×720
* תוכן סטטי: מסמכי Word, עורך קוד
* תנועה גבוהה: וידאו YouTube, משחקים

✅ **שליטה מרחוק:**

* תנועת עכבר: חלקה וללא שיהוי משמעותי
* לחיצות: Left/Right/Middle - עובד
* מקלדת: כל המקשים + modifiers
* Drag & Drop: עובד

✅ **תנאי רשת:**

* LAN (1 Gbps): מושלם
* WiFi (100 Mbps): טוב מאוד
* WiFi (10 Mbps): סביר (FPS נופל ל-15 עקב Adaptive Quality)
* Packet Loss 5%: Jitter Buffer מתמודד
* Packet Loss 15%: נראים artifacts, אך עדיין פועל

✅ **ביצועים:**

* CPU: 15-40% (תלוי ברזולוציה)
* RAM: ~200MB
* Network: 2-8 Mbps (תלוי ב-Motion Level)
